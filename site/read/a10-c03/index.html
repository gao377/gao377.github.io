消息已经送达，为什么事情仍没有发生——消息模式与动作生命周期

身份与版本对齐后，我终于能确认：启动请求确实被设备接口收到。传输记录写着 acknowledged，排水渠巡检设备那边的一条阀门核对请求也显示 delivered。

可 delivered 只停在“消息到了”。阀门有没有动作、泵站接口有没有施加启动、取消是否赶在动作以前，记录都没有回答。若我因为超时再发一条“内容相同”的新命令，两条迟到消息可能让不可逆动作执行两次。

我先把“送到”“接受”“做了”“得到结果”分开。博士随后给四种常见交互命名：连续更新是 stream，一次状态变化是 event，有界问答是 request/response，需要反馈、取消和最终结果的长过程是 long-running action。让重复请求只对应同一次逻辑动作的性质叫幂等性。

交互方式来自事情怎样持续

模式 适合的对象 必须保留的语义 不能从传输推出什么
stream 状态、观测、连续 telemetry sequence、sample time、revision、latest/queue 每个样本都被处理
event 接触变化、急停、版本切换 event identity、去重、因果引用 丢一条也无影响
request/response 查询状态、清理缓存、幂等配置 request ID、超时、领域结果 收到响应就代表物理成功
long-running action 启动、阀门动作、导航或安装 goal、feedback、cancel、applied、outcome goal accepted 就代表完成

接口不是按消息大小选择的。同一份 payload 若表达连续状态，应是 stream；若表达不可重复的状态转移，就必须有 event 或 action 身份。把长动作塞进短请求，会失去反馈和抢占；把关键事件放进“只留最新”的状态流，又可能被后来样本覆盖。

一个动作有自己的生命史

同一 action 至少沿这些状态前进：

提出动作 → 收到请求 → 接受请求 → 正在执行 → 已经施加 → 得到结果
                         ↘ 已取消 / 已抢占 / 已失败

实际系统还要记录取消请求、取消是否被接受，以及取消到达时动作是否已经 applied。cancel_sent 只是又一条 commanded 事实；它不能倒写历史。

对动作 aaa，完整身份可写成

Ia=(participant,task,resource,action_id,authority_epoch).(1)I_a=(participant,task,resource,action\_id,authority\_epoch). \tag{1}Ia​=(participant,task,resource,action_id,authority_epoch).(1)

状态转换只有在这个身份不变时才属于同一次动作。换一个 action_id，即使 payload 一字不差，也是在创建新的物理意图。

幂等不是“随便重发都安全”

一个幂等处理器应先按 action_id 查账：

收到 action_id
若从未见过：核对权限和前置，记录 received/accepted，再决定是否施加
若已经见过：返回同一动作的已知状态，不再次施加物理动作
若账本缺失或结果不确定：进入查询/核对，保持 UNKNOWN
取消到达：判断动作尚可抢占、已经 applied，还是终态已发生

这仍需要持久状态和恢复合同。设备重启后忘记去重账本，所谓幂等就只剩接口承诺。相反，即使传输系统“至少送达一次”，领域层仍可借稳定 action identity 避免重复物理动作。

“结果未知”不是失败的委婉写法

若 action 已 accepted 后断链，本地既不能宣布成功，也不能用超时宣布失败。远端可能尚未施加，也可能已施加但结果回报丢失。当前可确认的是通信结果未知；下一步应查询同一 action、保持本地安全，必要时抢占或停止依赖它的后续任务。

这也是排水渠巡检设备必须保留本地保护的原因。我的网络状态不能替它判断水渠内是否安全，它的 accepted 也不能替我按下人类尺度面板。

练习

练习 1：delivered 以后没有结果

启动消息 delivered，设备接口没有 accepted 或 applied 记录。判断能否创建一条内容相同的新 action，当前应查询、重发同一身份、保持还是宣布失败。

练习 2：取消可能来晚了

一条阀门 action 已 accepted；取消消息 sent，随后链路中断，applied 状态未知。说明本地能否假定阀门未动作，以及后续试运转应怎样处理。

练习 3：状态流覆盖了急停解除事件

系统用 latest-only stream 表示急停状态。解除事件短暂出现后被新样本覆盖，审计记录中没有 event identity。决定这条信息是否足以重新启动。
