第十幕

排水渠巡检设备断线了，我还知道任务做到哪里吗——任务身份、限时授权与间歇通信

泵站试运转前，我需要确认进水渠和蓄水池阀门的局部状态。第七幕留下的排水渠巡检设备仍保持原来的身体与职责：低矮、防水，能进入我不该进入的渠内，并在本地保护条件下核对阀门；它不能伸手操作人类尺度的控制面板，那仍是我的身体和本地权限范围。

它接受了一条阀门核对 action，链路随后中断 1.5 s。我的记录停在 accepted。如果我把超时当作失败并重发，阀门可能已经动作两次；如果我把 accepted 当作完成，泵又可能在前置未成立时启动。

我先问：当彼此不再能听见时，我们之间还剩下哪些可确认的关系？博士这才引入协作接口、authority epoch、lease、幂等 action与reconciliation。协作不是把两个身体接成一个进程，而是让各自的身份、能力、本地保护和未知结果在连接变化时仍不被抹掉。

先保留参与者的差异，才谈协作

协作登记中，每个 participant 都有稳定身份、boot/epoch、身体能力、资源 owner、本地安全边界和可授权动作。当前关系至少有四类不同实体：

  • 我：人形身体，位于泵站控制室，负责人类尺度面板与本地受控动作；
  • 排水渠巡检设备：低矮防水身体，负责本地水渠、阀门观察与局部保护；
  • 城市保障运输设备：承担运输，不是巡检设备的另一个名称；
  • 基础设施节点：提供泵站或城市设施状态，不是具身执行者的替身。

通信拓扑描述某一时刻谁能与谁交换何类消息，并不把节点身份合并。邻居离线、路由改变或经中继通信，都要产生 topology revision；旧拓扑下取得的权限不能默默沿用到新拓扑。

一项协作动作要能从任务追到物理结果

仅发送“检查阀门”不够。消息至少关联：

Ia=(Itask,Iresource,Iaction,Iowner,eauth,Iattempt).(1)I_a=(I_{task},I_{resource},I_{action},I_{owner},e_{auth},I_{attempt}). \tag{1}Ia​=(Itask​,Iresource​,Iaction​,Iowner​,eauth​,Iattempt​).(1)

任务、资源、动作、owner、授权 epoch 和尝试身份分开后，系统才能判断两条相似消息是同一 action 的重传，还是一个明确的新任务。

动作状态仍沿用本幕前面的语义：

提出动作 → 收到请求 → 接受请求 → 已经施加 → 得到结果

accepted 只证明远端在当时的前置下接纳任务；不证明已施加，更不证明阀门达到目标。断链时最后确认状态与当前真实状态之间会张开一个未知区间，不能用超时把它补成 success 或 failure。

限时授权只约束权限寿命，不报告动作结果

授权者可在 authority epoch eee 下发放从 tissuet_{issue}tissue​ 到 texpiryt_{expiry}texpiry​ 的 lease：

L=(p,r,c,e,tissue,texpiry),(2)L=(p,r,c,e,t_{issue},t_{expiry}), \tag{2}L=(p,r,c,e,tissue​,texpiry​),(2)

其中 participant ppp 在资源 rrr 上只拥有能力集合 ccc 内的权限。当前时刻落在 lease 内，只说明授权尚未按时间过期；它不说明链路健康、动作已执行或结果已知。

本幕固定场景中，链路中断 1.5 s，lease 长 3.0 s。即使中断时仍在 lease 内，远端 action 的 outcome 仍应保持 UNKNOWN。本地不能用“还没到三秒”推断远端正在正常执行，也不能在同一资源上另发可能重复的不可逆动作。

epoch 用来阻止旧权限复活。重新授权后，延迟到达的旧 epoch 消息即使通过可靠 QoS 完整送达，也必须被拒绝并进入审计；可靠交付不能把过期 authority 变成当前 authority。

幂等需要动作协议支持，不是把文本再发一遍

幂等 action 表示同一 action ID 的重复提交不会重复产生物理效果。远端要持久保存去重与结果账本，收到重复 ID 时返回原状态或继续同一事务，而不是重新动作。

有些物理动作天然不幂等，例如“再转动阀门一定角度”。可以把协议改为目标状态加前置与资源 revision，例如“在阀门 revision rrr 下到达位置 q∗q^*q∗”，但执行端仍需检测当前位置、限制作用并记录 applied/outcome。仅让消息内容相同，不会自动得到幂等性。

取消也遵循同样边界：cancel requested 不等于动作撤销。若远端可能已经 applied，本地要保留 UNKNOWN 并查询，而不是用新 action 覆盖。

断链状态机先保护两端的本地自治

协作链可以用四个状态组织：

状态 可以做什么 不能假定什么
CONNECTED 在身份、权限、freshness 和本地前置下交换任务 连接永久持续
DEGRADED 减少消息、只保留关键健康/结果/停止查询 普通遥测仍足够新
DISCONNECTED 各自执行预先声明的本地保护，冻结远端结果 对方成功或失败
RECONCILING 用原 action ID 查询、比对账本和状态 一收到消息就恢复权限

断链时，巡检设备继续遵守自己的渠内保护，不因本地消息消失而越界；我则根据最后确认的远端 health、样本年龄和任务依赖选择等待、HOLD 或中止。本地可继续与远端职责无关的安全工作，但不能进入依赖其局部保护的低处，也不能启动依赖阀门确认的泵任务。

重连后先认证 participant 和 boot/epoch，再交换 last-confirmed action、resource revision、health 与 outcome。若两边账本一致，闭合同一 action；若冲突，保持资源冻结并交由声明的 owner/人员裁决。只有 reconciliation 完成后，才决定继续原任务、重新授权或创建身份明确的新任务。

传输承诺与安全分别回答不同问题

  • authentication 证明消息来自谁，不证明其有权操作该资源；
  • authorization 判断该身份在当前 epoch/lease 下能做什么；
  • encryption/confidentiality 防止内容被未授权读取，不证明内容正确；
  • integrity 检测消息被篡改，不证明传感器没有故障；
  • audit 保存谁在何时请求、允许、拒绝和实施了什么；
  • QoS 管理交付、历史、期限与 liveliness，不证明 freshness 或物理 outcome。

远端 action 在进入 accepted 前，必须同时通过身份、授权、epoch、lease、resource revision、QoS/freshness 和本地保护。把认证成功直接当作授权，是把“我知道你是谁”误写成“你可以做任何事”。

事件触发通信还不是事件触发控制

为了节省带宽，可以规定只在健康变化、lease 临近到期、action 转态、序号间隙或查询触发时发送某类消息。这是事件触发通信：它要说明触发条件、最大静默期、freshness 与漏发处置。

若进一步让控制输入只在误差事件发生时更新，就进入事件触发控制；还需证明闭环触发规则、可行性、稳定性、执行更新和网络异常下的行为。本章不提前推导它。图 Laplacian、共识、刚性、编队与 distributed MPC 也留到第十二幕，它们不能替代这里的身份、权限和结果语义。

练习

练习 1：accepted 后断链，lease 尚未过期

远端 action 已 accepted，链路中断 1.5 s，lease 为 3.0 s，health 与 outcome 均无法刷新。本地试运转正在等待。决定能否重发、继续、取消、等待或停止，并说明 outcome 如何记录。

练习 2：可靠消息来自旧 authority epoch

重连后，一条完整且签名有效的旧 epoch applied 消息晚到；当前资源已经重新授权。判断应接纳、拒绝还是据此改写结果，以及如何审计和 reconciliation。

练习 3：只有状态变化才发送消息

团队计划让远端健康“没有变化就不发送”，并把这称为事件触发控制。指出当前真正设计了什么，还缺哪些静默上界、freshness 和闭环证明。

参考答案

重连后，巡检设备用同一 action ID 确认阀门动作已经 applied，资源 revision 与本地状态也闭合。我终于可以在人员许可和其余前置成立时，面对一次低能量的受控试运转。
