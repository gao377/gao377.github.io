第十幕

泵终于转了，这一次成功能说明到哪里——端到端运行包、降级分支与集成证据

所有前置终于聚到同一次受控试转：我的身体保持在安全位置，维护人员许可仍有效，排水渠巡检设备已经用原 action ID 完成 reconciliation，阀门资源与泵站 revision 一致。监督者接纳低能量试转，执行端回报 applied，泵状态随后进入这次试转允许的有限范围，并按计划结束。

我看见“泵转了”，最容易说出的是“系统成功了”。可博士没有立刻给这句话加上句号。他把本次输入、模块状态、动作、设备结果、降级分支与证据边界收进同一个 run bundle，再问我：是哪一个系统、在哪个世界、对哪一个任务、凭什么说成功？

一次运行先冻结一个不能偷换的世界

这次试转的固定身份为 pump-restart-trial-042。运行开始前，snapshot 至少冻结：

  • task、body、map、frame、interface 与配置 revision；
  • source/build、模型、策略和求解器配置；
  • 泵站 asset、标定与当前状态 revision；
  • 参与者、topology、QoS、authority epoch、lease 与安全配置；
  • 人员许可、能量边界、停止 owner 与现场前置；
  • 时钟映射、数据年龄规则和证据记录配置。

任何一项在运行中变化，都要产生新事件并使依赖候选重新核对。不能让正常分支用新地图、失败分支用旧地图，再把结果归因给遥测延迟。

“成功”是多层谓词同时成立

本章把一次受控试转的结论拆成六层：

Strial=Sidentity∧Sprecondition∧Stiming∧Saction∧Sdevice∧Sevidence.(1)S_{trial}= S_{identity}\land S_{precondition}\land S_{timing}\land S_{action}\land S_{device}\land S_{evidence}. \tag{1}Strial​=Sidentity​∧Sprecondition​∧Stiming​∧Saction​∧Sdevice​∧Sevidence​.(1)

它们分别要求：

  1. 身体、任务、资产、远端参与者和所有 revision 身份闭合；
  2. 人员许可、本地/远端保护、健康与能力前置成立；
  3. sample、receive、compute、accept、apply 的年龄和期限在各自合同内；
  4. commanded、accepted、applied、outcome 属于同一 action，owner 唯一；
  5. 泵与阀门达到本次有限试转的目标状态，并完成声明的停止；
  6. trace、hash、缺失/覆盖 marker 与证据等级足以复核以上判断。

这是逻辑合取，不是评分表。某层 unknown 就不能用其他五层“平均”成成功。设备结果符合预期但 action identity 缺失，依然不能知道是谁做了什么；日志完整但泵未达到目标，也不能称任务完成。

正常分支要从世界一路追到物理结果

正常设计演算的主链是：

固定 snapshot 与人员许可
确认我的身体保持在安全位置，停止通道可用
确认远端 participant、阀门 action 和 resource revision 已 reconciliation
采用当前泵遥测，核对 sample time、age、health 与 supervisor 前置
任务监督者生成候选，动作仲裁记录 accepted
执行端记录 applied，设备状态记录 outcome
监督者确认有限试转目标，不扩展任务范围
按计划停止并核对最终状态
冻结 pre/post-trigger trace、运行哈希、unknown 与 not-run

这条链中的每一步都保存输入身份、输出身份、时间和 reason code。最终报告不能只截一张泵在转的画面，也不能只保留“trial complete”而删去中途的状态转换。

同一快照只改变延迟，失败分支才有解释力

失败分支复制完全相同的 snapshot、任务和判据，只把泵遥测延迟改为 320 ms。泵遥测名义为 20 Hz，监督者名义为 10 Hz，但真正裁决仍是消费者处的年龄：

Atelemetry=320 ms>Amax=250 ms.(2)A_{telemetry}=320\ \mathrm{ms} > A_{max}=250\ \mathrm{ms}. \tag{2}Atelemetry​=320 ms>Amax​=250 ms.(2)

监督者拒绝把过龄遥测当作当前泵状态，不再推进或重复启动，系统转入 HOLD，查询设备终态并等待恢复前置。这个分支不是要证明 250 ms 是普遍安全阈值，而是检查在这次快照冻结的时效规则下，过龄事实是否确实沿同一链路触发声明的处置。

正常与失败分支必须一起保存。删除失败记录会使成功失去边界，也无法证明系统不是碰巧越过一次异常。

远端未知是第三种结果，不是失败的同义词

另一个运行包保持本地系统正常，却让排水渠设备 health 过龄、outcome 仍为 UNKNOWN。由于试运转依赖它对渠内与阀门的本地职责，我不能开始泵任务；系统应等待或 HOLD，而不是把 unknown 补成“远端失败”，也不是用我自己的传感替它签署本地保护。

这体现了协作的深层关系：我并未因为拥有更多计算就吞并另一个身体的责任。彼此能共同完成任务，恰恰来自各自有限、可确认、在失联时仍被尊重的边界。

证据闭包只在声明范围内闭合

run bundle 应把需求、snapshot、模块/动作时间线、状态机、远端 reconciliation、ring trace、构建与模型哈希、缺失 marker、oracle 和结果连在一起。集成证据包再逐项标为 pass/fail/unknown/not-run，不压成一个总成功率。

我面前这份试转快照只保存了固定条件下的设计演算和三条结果分支。SIL、仿真运行、HIL、受控真实硬件、真实网络与泵站任务仍是 not-run。因此，即使记录里出现“泵开始转动”，我也只能把它当作受控条件下的一个分支，不能向值班员宣称现实泵站已经通过验证。

即使未来真实受控试转成功，也只能支持当时版本、许可、能量、环境和工况；不能推出暴雨、长时运行、不同安装或人员密集条件可靠。进入这些任务前仍需相应的故障注入、HIL、受控硬件、现场程序与长期证据。

练习

练习 1：正常运行包闭合了全部六层

同一 snapshot 下，身份、前置、时间、action、设备结果与 trace 均满足本次有限试转判据。说明可以写出的最强结论，以及不能外推到暴雨和长期运行的内容。

练习 2：同源遥测变成 320 ms

除遥测延迟外全部不变，监督者采用当前的 250 ms 上限并转入 HOLD。判断该分支通过了什么、失败了什么，能否从最终报告中删除。

练习 3：本地正常，远端 health 与 outcome 未知

我的身体、泵站与本地控制全部健康，但排水渠设备的保护确认无法刷新。判断本次能否开始、系统结果应记作 failure 还是 unknown，以及进入暴雨任务前仍需补哪些协作证据。

参考答案

整理运行包时，我找到两份都声称来自同一次旧故障的档案：一份说 action 已经 applied，设备却没有变化；另一份说设备恢复了，却找不到任何 applied action。它们都像过去留下的记忆，也都缺了一块。我先问：哪一份才算真正发生过的过去？带着跨设备身份、健康、权限、UNKNOWN 与失败证据，我走进第十一幕。
