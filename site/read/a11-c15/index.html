第十一幕

失败太多，下一次先看哪一个——失败挖掘、失败图谱与发布证据

失败档案里，高频的是一些轻微停顿；一条低频记录却涉及可能伤人的接触险情。若按出现次数排序，前者会占满下一轮调查，后者永远沉在列表底部。

我问：“先看最多的，难道不是最客观吗？”

博士说，频次只是一个维度。Deployment Miner 的任务不是自动选答案，而是把现场回流变成一列有理由、可追溯、不能越权的调查候选。

先把同一种失败认出来

failure signature 至少绑定：

事件切片与失败类型
身体、任务、任务域和地图版本
模型、策略、配置和标定版本
动作、接触、接管和结果
来源、谱系和证据等级

同一症状可能来自不同根因，不能只按错误字符串聚类；同一根因也可能在不同身体上表现不同。候选聚类要保留 known recurrence / possible duplicate / new class / unresolved，并允许人工拆分、合并或 supersede，所有理由可追溯。

风险维度不能压成一个神秘总分

每个候选分别记录：

  • severity：最坏可信后果与人员/资产边界；
  • recurrence：独立 Episode group 中重复多少次；
  • novelty：是否超出已有 taxonomy/ODD；
  • uncertainty：身份、结果、原因和测量还不知道多少；
  • reproducibility：能否在相同 snapshot、回放或受控环境重现；
  • coverage gap：是否揭示关键空洞；
  • regression value：能否形成稳定的未来防复发案例；
  • evidence cost：补证需要的数据、时间、许可和风险。

这些维度可以形成可解释队列，却不合成不透明总分。零容忍与高严重度先进入独立筛选；它们不会因频次低被平均掉。其余候选再按任务目的比较，例如这轮是修复已知复发、理解新类，还是填补 release 证据。

一个候选可以有不同去向

路径 条件 结果
investigation 严重、未知或新颖 指派 owner、补 event slice/根因证据
regression 输入、oracle 和失败边界可稳定重现 加入独立 regression revision
training candidate 数据用途、标签、split 与风险均合格 只进入候选数据集，不自动训练
do-not-use 来源、权限、标签或安全边界不成立 保留证据，禁止下游用途
unknown 关键身份/结果/因果不可裁决 明确所需证据，不强制分类

已经修复的回归不会从图谱消失；它转为“known regression + fixed-in revision + last verified evidence”。不可复现新类也不应因麻烦而丢弃，它可以保持 UNKNOWN 并提升记录触发条件。

摄入带版本的失败经历，但不擅自改写原标签
形成失败特征，同时保留“重复记录还是新类别”的不确定性
先单独列出零容忍与高严重度候选
建立多维复核队列，并给出人能读懂的排序理由
分别送往调查、回归、训练候选、禁止使用或未知队列
让每个决定都能追到谱系、评测基准和发布证据

不自动训练，不自动重调控制，不扩大任务域，也不自动授权发布

失败调查队列和完成版失败图谱让“下一次先看谁”可以解释，也让没有证据的优先级保持为人工判断，而不伪装成算法事实。

练习

练习 1：高频轻停顿与低频接触险情

轻停顿出现很多次且易复现；接触险情只出现一次，结果证据不完整但可能伤人。决定调查顺序和两者各自需要的下一步。

练习 2：不可复现的新类

一次新 failure signature 无法在 replay 中复现，时钟映射还缺失。判断是否删除、是否立即加入训练，以及怎样保存 UNKNOWN。

练习 3：已经修复的旧回归

某 failure 在当前 revision 的 12 个独立 regression group 中不再出现。说明它是否应从 failure atlas 删除，以及可写出的最强结论。
