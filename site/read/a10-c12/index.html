故障只出现一瞬间，怎样让它留下证据——环形缓冲、事件轨迹与可复现回放

一次短暂的队列拥堵只持续了几个周期：状态变旧，监督者进入 HOLD，随后数据又恢复。等我停稳后打开普通日志，文件里只有“进入保持”和一串恢复后的正常消息。拥堵前队列怎样增长、哪条样本被采用、谁先触发状态转移，都已经被新记录覆盖。

我需要的不是更多文字，而是故障前后仍能保持因果关系的证据。博士这才引入环形缓冲（ring buffer）、结构化 trace 与可复现回放：快路径只把有界记录写入预分配内存，触发后冻结故障前窗口并继续收集有限的故障后窗口，再由非关键路径异步保存。

环形缓冲先保住“发生前”

普通文件从故障发生后才开始写，往往来不及保存前因。容量为 NNN 的环形缓冲持续覆盖最旧记录，因此始终保留最近一段历史。若事件率上界为 rrr，希望保存触发前 TpreT_{pre}Tpre​ 与触发后 TpostT_{post}Tpost​，理想的最低记录槽数量满足

N≥r (Tpre+Tpost).(1)N\ge r\,(T_{pre}+T_{post}). \tag{1}N≥r(Tpre​+Tpost​).(1)

这个不等式先回答一个很朴素的问题：如果每秒最多会留下 rrr 条记录，而我想保住故障前后总共 Tpre+TpostT_{pre}+T_{post}Tpre​+Tpost​ 秒，就至少要为这段时间里可能出现的记录留出位置。它没有替我算出真实容量，因为泵站的突发事件率、单条记录大小和写出延迟还没有测得；它只是阻止我凭感觉说“缓冲区应该够大”。

真实设计还要加入突发率、多个事件类型、写出等待和安全余量；若 rrr 本身没有上界，式（1）也无法给出可信容量。记录大小可变时，还要按字节预算和最坏单条大小分析，不能只数消息条数。

触发流程可以是：

平时覆盖最旧记录，并维护 write sequence 与 high-water mark
触发发生时保存 trigger identity、触发者、原因与当前 write position
锁住触发前窗口的逻辑边界，继续收集有界 post-trigger 窗口
控制线程只提交已预分配记录，不等待文件、压缩或网络
低优先级 writer 异步复制并持久化证据包
写出失败、覆盖、丢项和截断必须显式留下 marker

“无日志”不能解释为“无故障”。若缓冲溢出、writer 失败或触发太晚，证据结论应标成 partial/unknown，并说明缺失范围。

事件轨迹记录身份，不只记录一句话

跨模块重建至少需要这些共同字段：

  • trace/run identity 与 trial identity；
  • participant、process、thread/executor、boot/epoch；
  • event identity、sequence、sample/receive/start/finish/apply time 与 clock domain；
  • body/map/task/config/model/policy revision；
  • topic/action/service、topology snapshot 与 QoS；
  • health、liveliness、progress、queue depth/high-water；
  • fault injection、状态机转移、reason code 与动作终态；
  • causal parent 或触发事件身份。

因果 parent 不是声称哲学上的唯一原因，而是记录“这个决定使用了哪些事件”。例如 HOLD 转移应关联过龄遥测判定，判定再关联其 sample time、消费者阈值和时钟健康。只有相同毫秒内的两行日志，不能证明谁导致了谁。

跨设备 trace 还要保留同步不确定度。若无法建立可靠的全局顺序，可以保留每个参与者的局部顺序和消息关联，明确哪些事件只能判“可能先后”，不能为了画出整齐时间线而伪造精确排序。

回放需要冻结产生结果的世界

一份可复现证据包不仅含输入流，还应保存：

  • 源代码或构建身份、依赖与编译配置；
  • 参数、环境、接口 schema 与 topology 快照；
  • 身体配置、地图、任务、模型与策略哈希；
  • 随机种子及随机数生成规则；
  • 初始状态、调度/网络故障注入与时钟映射；
  • 期望的关键事件、动作、reason code 和允许差异。

golden trace 不是一段永远正确的录像，而是一份经过审查、带来源和适用 revision 的比较基线。系统改变后，应判断旧基线仍适用、需要迁移还是必须退休，不能只更新期望输出让测试重新变绿。

回放也分层：消息按原顺序重新喂入，不能自动重现线程抢占；固定随机种子，不能自动重现 GPU 归约；仿真里重现状态机，不能自动证明真实泵或身体会有同样反应。因此每次回放必须写清重建了哪些因素、哪些时间关系只是近似。

一次只改变一个故障因素

若同时注入延迟、丢包、时钟漂移和模块重启，看到 HOLD 后很难知道哪项前置真正触发。诊断阶段应从同一基线复制运行，一次只改变一个变量，再逐步组合：

  1. 固定输入与构建，确认无故障基线可重复；
  2. 只注入一种延迟或序号重排；
  3. 比较首次分歧事件，而不只比较最终状态；
  4. 保存触发前后 trace 与运行身份；
  5. 单因素解释成立后，再测试组合故障与共同原因。

如果同一回放有时进入 HOLD、有时不进入，这本身就是结果：调度、未冻结随机性或数据竞争仍在证据模型之外，不能挑一次符合预期的运行作为结论。

练习

练习 1：故障发生前的记录总被覆盖

系统只在进入 HOLD 后开始写文件，因而没有队列增长过程。设计 ring buffer 的触发前/后窗口、快慢路径和溢出语义，不需要给出虚构的容量数字。

练习 2：两台设备的日志时间互相矛盾

我的日志写着先收到远端健康，排水渠设备日志却像是先断连；时钟同步当时不健康。说明仍可重建哪些顺序、哪些必须保持未知，以及 trace 应补哪些关联字段。

练习 3：一次注入四种故障后得到预期停止

测试同时加入网络延迟、丢包、时钟漂移和远端重启，系统最终停止。判断这能证明什么，如何重排实验以定位首次分歧并形成可复现基线。

参考答案

故障现在能够留下可追查的痕迹。但“留下一份日志”离“证明系统可以走向现实”仍隔着很多层。我需要一把尺子，分清纸面推演、回放、仿真、半实物与真实任务各自能够说明什么。
