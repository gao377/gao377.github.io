软件已经通过了，为什么还不能碰真实设备——回放、软件在环与硬件在环的证据边界

我用上一章保存的记录回放同一快照，状态机再次在遥测过龄时进入 HOLD。软件的决定与基线一致，我几乎把“可以接泵”写进试运转清单。可回放里的泵没有惯性、阀门没有卡涩、执行接口也不会因电气噪声丢掉一次施加。

另一个候选环境接入了真实控制器接口，画面与信号都像现场，却仍使用模拟地面、模拟人员和模拟负载。我先发现：测试通过只能回答它真正包含的问题。博士于是引入证据梯度（evidence ladder）、软件在环（SIL）、**硬件在环（HIL）**与 test oracle，让每一层都声明哪些部分真实、哪些部分模拟、依据什么判断对错。

证据层不是升级勋章，而是不同问题的边界

层次 主要包含什么 可以支持的结论 仍不能支持的结论
内容 / 静态检查 文本、schema、引用、固定算例 合同内部一致、字段齐全 软件能运行、物理任务可行
unit 独立函数或组件与人工 fixture 局部数值和边界行为符合 oracle 组件组合、真实时序与 I/O
integration / golden 多组件、冻结输入与期望事件 固定版本组合满足已覆盖路径 未覆盖输入与现实动力学
recorded replay 真实或合成记录再次喂给软件 对已记录输入的决定可重现到声明程度 新的闭环反应、调度确定性
SIL 真实候选软件，模拟 plant 与 I/O 软件闭环逻辑在模型范围内运行 真实执行器、接口硬件和人员风险
simulator study 明确保真度的模型与场景 在模型假设内比较策略、故障与趋势 未建模现实条件和真实成功率
HIL 某些真实控制器/接口/硬件，余下模拟 所含硬件边界、I/O 与部分时序表现 完整身体、地面、泵站与人员环境
controlled hardware 真实硬件、受限能量与受控场地 指定配置和工况下的物理观察 长期、生产或极端天气任务
task environment 真实任务环境、权限与运行条件 仅在声明任务域内形成现场证据 未覆盖季节、故障和未来版本

这些层次不必机械地逐级完成，也不能用一个总分相加。它们的价值在于把结论的谓词写完整：哪个构建、哪套身体和资产 revision、什么输入范围、何种环境、观察了几次、由哪个 oracle 判断。

判定准则决定“通过”究竟是什么意思

test oracle 是判断预期与实际是否一致的规则。它可以来自解析性质、独立参考实现、约束验证器、经审查的 golden trace、物理测量或人员明确判据。只比较“程序没有崩溃”是很弱的 oracle；只比较最终泵状态，也会漏掉中途重复动作与错误权限。

同一测试通常要同时观察：

  • 接口、单位、frame、revision 与身份是否匹配；
  • 数值是否在预先冻结的容差内；
  • deadline、data age、队列与 apply 是否满足；
  • action accepted/applied/outcome 是否闭合；
  • 降级、unknown 与停止分支是否按合同发生；
  • trace 是否完整，缺失是否显式标记。

oracle 自身也需要版本与适用范围。若系统更新后只是把 golden 输出替换成新结果，测试可能失去发现回归的能力。

低层通过不会自动传递到高层

unit 通过说明函数对所测输入正确，不推出 integration 通过；SIL 通过说明软件与模拟 plant 的闭环符合该模型，不推出真实接触正确；HIL 通过说明被纳入的硬件边界经受了已测条件，不推出人员和泵站负载已经覆盖。这种证据不传递性不是否定低层测试，而是阻止结论越过测试真实包含的边界。

反过来，真实硬件偶然成功也不能替代低层证据。若没有可复现输入、trace 和故障注入，现场异常难以定位，版本更新也无法做便宜而频繁的回归。证据梯度需要上下互相约束，而不是只追求最高一层的演示。

一份证据包应让未知和未运行留下位置

针对 pump-restart-trial-042，每一项 requirement 应关联测试身份、输入 snapshot、真实/模拟边界、oracle、预期结果、实际结果和证据位置。结果至少区分：

  • pass：在声明范围内得到并满足判据；
  • fail：得到结果但违反判据；
  • unknown：运行过，证据不足以判定；
  • not-run：尚未执行这一层。

unknown 不能补成 pass，not-run 也不是 fail。覆盖率只说明被测结构或需求比例，不说明未覆盖部分安全；故障注入通过只覆盖所注入的故障模型，不能泛化为“能处理所有故障”。

眼前能够形成的是一份纸面证据梯度与待执行证据包。SIL、仿真、HIL、受控硬件和真实任务环境都保持 not-run。因此现实能量隔离仍然成立，记录里出现泵站名称也不表示真实设备已经连接。

练习

练习 1：golden replay 再次进入正确状态

固定记录回放后，软件再次在过龄遥测时进入 HOLD，全部 reason code 与基线一致。判断它能说明接口、数值、时序、物理泵和人员安全中的哪些，证据包应怎样写。

练习 2：SIL 中完成了泵重启流程

候选软件与模拟泵、模拟 I/O 闭环运行，正常与遥测延迟分支都符合 oracle。说明可以提升哪些软件结论，哪些现实结论仍是 not-run。

练习 3：HIL 接入真实控制器接口

HIL 包含真实控制器和接口板，但身体、地面、泵负载与人员都模拟。决定能否解除现实设备隔离，并列出继续前必须补齐的边界声明。

参考答案

证据边界划清以后，我看到一个据称会随现实泵站持续变化的模型。画面极其逼真，但它连现场正在使用哪一版阀门配置都不知道；这样的模型，真的已经是“孪生”吗？
