第十幕

地图已经更新，旧计划为什么还在向前走——版本传播、生命周期与抢占

排水渠侧的新观测让地图向前更新了一版，定位结果和坐标变换也随之改变。可任务面板仍把旧 plan 标成 active，控制器缓存着旧 transform，已经 accepted 的动作继续等待施加。

我原以为“发布新地图”会让所有模块自然读取新值。实际发生的是：新数据已经存在，旧动作却没有收到停止的事实。博士这才引入依赖版本、生命周期（lifecycle）、抢占（preemption）和原子激活：更新一个输入，只会让依赖它的旧结果失效；要停止旧动作、清理缓存并启用新快照，还需要明确的状态转换。

计划保存的是一组依赖，不是一个文件名

设一条计划产生时绑定的快照为

Splan=(rbody,rmap,rtf,rtask,rconfig).(1)S_{\mathrm{plan}}= (r_{\mathrm{body}},r_{\mathrm{map}},r_{\mathrm{tf}}, r_{\mathrm{task}},r_{\mathrm{config}}). \tag{1}Splan​=(rbody​,rmap​,rtf​,rtask​,rconfig​).(1)

当前系统也有一个版本向量 ScurrentS_{\mathrm{current}}Scurrent​。只有相关分量一致，计划才可能继续满足原前置：

valid⁡(plan)=⋀j∈deps⁡(plan)rjplan=rjcurrent.(2)\operatorname{valid}(plan) = \bigwedge_{j\in\operatorname{deps}(plan)} r_j^{\mathrm{plan}}=r_j^{\mathrm{current}}. \tag{2}valid(plan)=j∈deps(plan)⋀​rjplan​=rjcurrent​.(2)

式（2）只是版本必要条件，不证明几何、动力学或安全仍可行。反过来，发现版本不同也不会让执行器自动停下；它只产生“旧候选不能再被采用”的系统事件。

变换必须属于测量发生的时刻

一条在 tst_sts​ 采样的观测，应使用该时刻有效的变换 mapTbody(ts){}^{map}T_{body}(t_s)mapTbody​(ts​)。查询“最新变换”可能把未来身体姿态用于过去观测。若所需时刻超出缓存，可以等待有界时间、丢弃该观测或降级；不能无声明外推。

地图 revision 更新还可能改变变换树的来源。一个 child frame 同时由两个 owner 发布，即使数值相近，也会让计划依赖不再可复现。更新前先确定唯一 owner，再谈数值。

进程存在不等于模块可以工作

本章用四个主状态描述模块：

状态 可以做什么 不能做什么
unconfigured 保留身份，等待配置 产生任务级输出
inactive 资源和参数已核对 接受并施加新动作
active 在当前快照内工作 跨失效 revision 继续
finalized 保存结束与故障证据 自动返回 active

地图、frame、控制频率或身体配置变化通常不能在 active 时只改一个字段。模块应先停止接受新任务，退到 inactive，重建缓存和前置，再经监督者确认后激活。固定等待几秒不是依赖条件；真正的条件是上游状态、版本和健康已经成立。

更新要把旧动作安全地排空

检测 body/map/transform/task/config revision 变化
沿依赖图标记受影响的 plan、cache、controller 和 action
先阻止新的 accept，保存当前 action 状态
向可抢占动作发送 preempt/cancel，并等待有界确认
已 applied 或结果 UNKNOWN：转保持/停止并继续核对，不假定已撤销
模块退到 inactive，清理旧 cache，形成一致的新 snapshot
重新规划、复核前置，再原子激活新 revision
记录旧版何时失效、新版何时取得所有权

所谓原子激活，不是所有模块在同一 CPU 指令中切换，而是下游只能看到“完整旧快照”或“完整新快照”，不能看到地图新、plan 旧、cache 更旧的拼接状态。若切换中途失败，系统应回到声明的旧安全状态或保持，而不是让最后成功更新的模块决定整体版本。

抢占也有可能太晚

preempt requested 不等于动作已经停止。若旧动作尚未 applied，可以取消并确认终态；若已经 applied，恢复要依据设备状态处理后果；若 applied 未知，则保持 UNKNOWN 并转入安全核对。回滚配置也不能回滚已经发生的物理动作。

练习

练习 1：新地图、旧计划、旧缓存

map 与 transform 已更新，plan 仍 active，controller cache 仍旧。安排停止接受、抢占、清缓存、重规划和恢复的顺序。

练习 2：抢占请求没有确认

旧动作已 accepted，preempt sent 后接口无响应，applied 状态未知。判断能否直接激活新 plan。

练习 3：只改版本号能否完成迁移

工程人员确认旧 plan 的几何“看起来仍能走”，于是准备把其 map revision 改成当前值。说明这样做破坏了哪些来源和复核关系。
