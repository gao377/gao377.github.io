第十幕

每个模块都正确，泵站为什么仍然不工作——把系统接成同一件事

全书路线

替换部件已经安装，我仍是上一幕那具人形身体，双手已经退出设备，双脚保持在泵站控制室的安全位置。可泵没有因此转动：身体控制说动作完成，消息模块说命令送达，接口说请求已经接纳，设备状态却没有形成同一个结果。

这一幕不再给我增加新的身体部件，也不靠重复安装或反复发送命令猜测根因。我先追问同名数据是否真是同一件事、消息何时发生、旧计划为何仍在执行、哪个模块拥有最后动作。博士随后才引入数据合同与版本、动作生命周期、时钟关系、传输承诺、多速率、实时调度、独立监督与降级状态机。

当系统能够保存瞬时故障，我们再区分内容回放、软件在环、硬件在环与真实任务的证据范围；数字孪生先绑定具体资产，从仿真走向现实的检查再逐项记录几何、接触、传感、时间、计算和通信差异。最后，第七幕留下的排水渠巡检设备重新进入任务：它仍保持低矮防水的身体、本地水渠责任与保护边界，不与运输设备或基础设施节点混为一体。

每一部分都完成，为什么没有同一个结果

  1. 每个模块都说完成了，泵为什么没有转
  2. 名字相同的数据，为什么不是同一件事
  3. 消息送达了，为什么动作仍没有发生
  4. 地图已经更新，旧计划为什么还在向前走

第一节先把模块局部完成与端到端系统结果分开。随后，接口身份不只是一串字段名，还包括单位、坐标系、数据形状、语义、来源与结构版本；同一动作也必须区分提出、收到、接受、施加和得到结果。

当地图、身体、任务或配置改变，依赖旧快照的计划、缓存和已接受动作不会自动消失。生命周期、抢占与原子激活让系统只看见完整旧版或完整新版，不把新地图、旧计划和更旧缓存拼成一个貌似正常的世界。

数据有多旧，快慢模块怎样共享同一个现在

  1. 四个时间，哪一个才是这条数据的现在
  2. 后到的数据，一定比先到的更新吗
  3. 快慢不同的模块，怎样共同控制同一具身体
  4. 同一个公式换一种实现，还是同一个结果吗

一条状态至少经历采样、接收、计算和施加；只有先保留时钟域、启动时期与同步不确定度，消费者才能正确判断数据年龄。可靠传输不等于及时可用，状态快照、动作事务、停止事件和诊断轨迹需要不同的队列与传输承诺。

全身控制、状态估计、策略、触觉和视觉以不同频率工作。零阶保持、插值和短时外推都只在声明的连续区间内成立；接触、版本、策略切换和停止是不能跨越的离散屏障。实现替换也不只比较浮点误差，还要比较约束决定、后备动作、判定原因、内存所有权与执行行为。

最慢一次赶不上时，身体由谁接管

  1. 最快的一次运行，能说明最慢的一次吗
  2. 计算来不及完成时，系统应该留下什么动作
  3. 一个模块沉默了，谁会先发现
  4. 故障只出现一瞬间，怎样让它留下证据
  5. 软件已经通过了，为什么还不能碰真实设备

0.7 ms 与 5 ms 只是一份给定运行样本中的执行时间，2 ms 与 20 ms 是设计期限；它们都不是已经证明的最坏情况执行时间（WCET）。任务模型、响应时间、阻塞与优先级反转说明平均很快为何仍可能错过关键时刻，端到端检查还要从输入的采样时刻一直追到动作真正生效。

超时以后，旧候选只有在版本、年龄、约束、接触模式和剩余时域仍成立时才能有界沿用。NOMINAL、CONSTRAINED、DEGRADED、HOLD、SAFE_STOP、E_STOP_LATCHED 与 RECOVERY_PENDING 分别对应可观察的身体行为；恢复不会因心跳重新出现而自动发生。

心跳也不能只证明进程存在。监督者要看关键功能的完成序号、及时性、版本和结果状态，并避开与被监督者相同的锁、调度与停止故障。环形缓冲保存触发前后，结构化事件轨迹保留参与者、时间、传输状态、健康、动作与因果关联；证据梯度再把纸面推演、回放、软件在环、仿真、硬件在环、受控硬件和真实任务分别标为通过、失败、未知或尚未运行。

数字世界怎样与现实资产和另一个身体相连

  1. 会变化的数字泵站，和普通仿真有什么不同
  2. 仿真里成功，哪些东西能跟着来到现实
  3. 排水渠巡检设备断线了，我还知道任务做到哪里吗
  4. 泵终于转了，这一次成功能说明到哪里

数字孪生先回答“对应哪一个现实资产和版本”，再谈模型多精细。状态同步必须保留采样时刻、不确定度、过龄与未知；模拟默认值不能冒充现场真值，数字模型给现实系统的建议仍要经过系统仲裁和人员许可。

从仿真走向现实也不使用一个总差异分数。几何、动力学、接触、传感、时间、计算和通信逐项记录为测量、辨识、随机化、假设或未知；生成数据还要保存生成器、场景、随机种子、参数、后处理、标签和下游用途的谱系。仿真、硬件在环和现实成功是三个不同谓词。

远端设备接受任务后断链 1.5 秒，而限时授权为 3.0 秒，只能说明权限尚未按时间过期，不能说明动作成功或失败。断链期间结果保持 UNKNOWN；重连后用同一动作身份、授权时期、资源版本与两端账本完成核对。事件触发通信在本幕只约束何时发送和查询，图拉普拉斯矩阵、共识、刚性、编队、分布式模型预测控制与事件触发控制留到第十二幕。

最后，同一个 pump-restart-trial-042 快照保存三条分支：正常试转；遥测延迟满足 320 ms > 250 ms，因而进入 HOLD；远端健康为 UNKNOWN。一次有界成功不能扩展成暴雨、长时或未来版本可靠；失败、未知与尚未运行也不能从最终记录里删除。

返回第九幕：我让双脚、双手和身体共同完成安装

整理运行包时，我找到两份都声称来自同一次旧故障的档案：一份记录 action 已经 applied，设备却没有变化；另一份记录设备恢复，却缺少 applied action。哪一份才算真正发生过的过去，将成为第十一幕的第一个问题。
